## Overview

The objective of this lab was to implement a multitasking embedded control system for the Romi robot using cooperative scheduling in MicroPython. Instead of running a single sequential program, the system was divided into multiple independent tasks responsible for motor actuation, encoder measurement, control computation, and user interaction.

The multitasking framework was built using a cooperative scheduler provided in `cotask.py`. Each subsystem was implemented as a task that runs periodically and communicates with other tasks through shared variables and queues. This approach allows the robot to simultaneously control motors, read sensors, execute control algorithms, and interact with the user through a serial interface.

This architecture represents a major step toward building a complete embedded robotics control system and serves as the foundation for future labs involving closed-loop velocity control and autonomous behaviors.

## System Architecture

The system was divided into multiple independent tasks that run under a cooperative scheduler.

The primary tasks implemented in this system were:

| Task | Responsibility |
|-----|-----|
| Motor Task | Reads encoder data and actuates motors |
| Control Task | Computes control effort using a proportional controller |
| User Task | Handles serial commands and data output |
| Scheduler | Executes tasks based on priority and timing |

Each task is implemented as a **generator function** that yields control back to the scheduler after completing a small amount of work. The scheduler repeatedly runs tasks based on priority and timing constraints.

---

## Cooperative Multitasking

The multitasking framework is implemented using the `cotask` module. Tasks are created as generator functions and added to a task list that the scheduler manages.

Each task runs for a short period of time and then yields control so other tasks can execute.

Example task creation:

```python
task_list.append(Task(leftMotorTask.run, name="Left Motor", priority=2, period=10))
```

The scheduler then runs continuously:

```python
while True:
    task_list.pri_sched()
```

This design allows multiple subsystems to operate concurrently without requiring a full real-time operating system.

## Task Communication

Tasks communicate using **shares** and **queues**.

### Shares

Shares store a single shared variable that multiple tasks can access safely.

Examples of shared variables in this system include:

| Share | Purpose |
|------|------|
| motorGo | Enables or disables motor control |
| setpoint | Desired motor velocity |
| Kp | Control gain |
| omega | Measured motor speed |
| effort | Motor command output |

Shares allow tasks to exchange data without risking memory corruption due to simultaneous access.

---

### Queues

Queues store a sequence of values and are used for logging experimental data.

For example:

| Queue | Purpose |
|------|------|
| dataValues | Stores measured motor velocity |
| timeValues | Stores timestamps for collected data |

Queues allow one task to generate data while another task reads and processes that data later.

## Motor Task

The motor task is responsible for controlling the motor and reading encoder feedback.

Its main responsibilities include:

1. Updating the encoder measurement  
2. Calculating motor velocity  
3. Sending measured velocity to the control task  
4. Applying motor effort commands  
5. Logging velocity and time data during experiments  

Example encoder update:

```python
self._enc.update()
omega = self._enc.get_velocity()
self._omega_meas.put(omega)
```

The task also stores collected data into queues for later analysis.

---

## Control Task

The control task computes the motor effort required to achieve a desired velocity setpoint.

A proportional controller is used:

u = Kp (r − y)

where

- r = reference velocity (setpoint)  
- y = measured velocity  
- Kp = proportional gain  

The control effort is then limited to prevent commanding excessive motor power.

Example controller update:

```python
effort.put(ctrl.update(ref, meas))
```

This allows the control task to continuously adjust motor effort based on measured velocity.

## User Interface Task

The user task provides a simple serial interface for controlling the system and collecting data.

The user can enter commands through the USB serial terminal:

| Command | Action |
|------|------|
| l | Start data collection for left motor |
| r | Start data collection for right motor |
| x | Stop motor operation |

When data collection is complete, the system prints the collected data in CSV format.

Example output:

```
Time, Omega
1234,5.21
1260,5.34
1290,5.28
```

This data can then be exported and analyzed using external plotting tools.

## Data Analysis

After collecting velocity data from the motors, the results are plotted using Python scripts.

Example plotting workflow:

1. Export CSV data from the serial terminal  
2. Run the plotting script  
3. Generate a velocity vs. time plot  

Example plotting code:

```python
plt.plot(time, omega)
plt.xlabel("Time (s)")
plt.ylabel("Omega")
```

These plots allow visualization of motor response and controller behavior.

## Discussion

This lab introduced an embedded multitasking architecture that allows multiple control subsystems to operate concurrently. By separating functionality into independent tasks, the program becomes easier to understand, modify, and expand.

The cooperative scheduler allows tasks to share processor time efficiently while maintaining predictable execution timing. The use of shares and queues provides a safe mechanism for exchanging data between tasks without introducing race conditions.

This architecture closely resembles the design of larger embedded robotics systems and prepares the platform for more advanced control strategies.

## Conclusion

This lab successfully implemented a multitasking control system for the Romi robot using cooperative scheduling in MicroPython.

Multiple tasks were created to manage motor actuation, encoder measurement, control computation, and user interaction. These tasks communicate through shared variables and queues while being coordinated by a priority-based scheduler.

The resulting system demonstrates how complex embedded control systems can be built by combining modular drivers, task-based software architecture, and cooperative multitasking.
