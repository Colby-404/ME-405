## Overview

The objective of this lab was to implement and evaluate a closed-loop motor velocity controller on the Romi robot using a multitasking architecture. The system used encoder feedback to measure motor velocity and applied a PID-style control algorithm to regulate motor speed to a specified setpoint.

The control system was implemented using cooperative multitasking in MicroPython. Separate tasks were used for motor actuation, control computation, user interaction, and system scheduling. This modular architecture allows each subsystem to run concurrently while maintaining predictable timing behavior.

Step response data was collected for the left and right motors using an automated data collection interface. The resulting data was exported in CSV format and analyzed to evaluate controller performance, including overshoot, settling behavior, and steady-state error.

The hardware platform used for this lab was the Pololu Romi robot configured with DC motors, quadrature encoders, and an STM32 Nucleo-L476RG microcontroller running MicroPython.

PWM signals generated by Timer 4 were used to control motor effort. The quadrature encoders were connected to hardware timers configured in encoder mode to measure motor rotation and velocity.

## System Architecture

The control system was implemented using a modular task-based architecture. The system is divided into several independent tasks that communicate using shared variables and queues.

| Task | Responsibility |
|-----|-----|
| Motor Task | Reads encoder data and actuates motors |
| Control Task | Computes control effort using PID control |
| User Interface Task | Handles commands and prints experimental data |
| Scheduler | Runs tasks cooperatively based on priority |

Each task is implemented as a generator function that yields control back to the cooperative scheduler. The scheduler repeatedly runs tasks according to their priority and period. :contentReference[oaicite:1]{index=1}

This architecture allows multiple subsystems to operate concurrently while maintaining predictable timing behavior.

## Encoder Measurement

Motor position and velocity are measured using quadrature encoders. The encoder interface uses a hardware timer configured in encoder mode to track wheel rotation.

The encoder driver updates position and velocity using microsecond timing resolution.

Example velocity calculation:

```
velocity = (delta_counts / dt) * 1e6
```

where

- `delta_counts` is the change in encoder counts
- `dt` is the time difference in microseconds

This approach provides high-resolution velocity measurements suitable for closed-loop control. :contentReference[oaicite:2]{index=2}

## Motor Driver

Motor actuation is handled by a dedicated motor driver class that interfaces with the DRV8838 motor driver hardware.

The motor driver accepts effort commands ranging from −100 to +100 percent and converts them into PWM duty cycles and direction signals.

Example motor command:

```
motor.set_effort(effort)
```

The driver also includes enable and disable functionality to place the motor driver into sleep mode when not in use. :contentReference[oaicite:3]{index=3}

## Closed-Loop Control

Motor velocity is regulated using a PID-style controller implemented in a dedicated control task.

The controller computes motor effort using the following equation:

u = Kp·e + Ki∫e dt + Kd(de/dt)

where

- e = reference velocity − measured velocity  
- Kp = proportional gain  
- Ki = integral gain  
- Kd = derivative gain  

The controller output is saturated to prevent commands exceeding the allowable motor effort range.

### Derivative on Measurement

An important design feature of the controller is the use of **derivative on measurement** rather than derivative on error. This approach helps prevent derivative spikes when the setpoint changes suddenly.

Instead of computing:

```
d(error)/dt
```

the controller computes:

```
-(d(measurement)/dt)
```

This provides damping without causing large control spikes when the reference signal changes. :contentReference[oaicite:4]{index=4}

## Motor Task Behavior

The motor task is responsible for reading encoder data, applying motor commands, and optionally logging data during experiments.

The task operates as a finite-state machine with the following states:

| State | Description |
|------|-------------|
| INIT | Initialize hardware |
| WAIT | Wait for control enable command |
| RUN | Execute closed-loop motor control |

The motor task continuously updates encoder measurements and applies effort commands from the controller.

A notable design feature is that **motor control and data logging are separated**. If the data logging buffers become full, logging stops but motor control continues uninterrupted. This prevents data collection from interfering with motor operation. :contentReference[oaicite:5]{index=5}

## Data Collection and Analysis

Step response experiments were conducted by commanding the motors to a specified velocity setpoint and recording the resulting velocity response.

The collected data was processed using a Python script that communicates with the microcontroller over a serial interface. The script automatically performs experiments, logs the resulting data, and generates response plots. :contentReference[oaicite:7]{index=7}

## Discussion

This lab demonstrates the implementation of a complete embedded control system using a modular multitasking architecture. By separating functionality into independent tasks, the system becomes easier to maintain, debug, and extend.

Several design decisions improve the robustness of the system:

- derivative on measurement prevents derivative kick  
- motor control remains active even if logging stops  
- shared variables provide safe inter-task communication  
- cooperative scheduling allows deterministic execution timing  

This architecture resembles the design patterns used in real robotic control systems and provides a strong foundation for implementing more advanced control strategies.

In this lab, a closed-loop motor velocity control system was successfully implemented using encoder feedback and a PID controller running within a multitasking embedded architecture.

The system was able to regulate motor speed to a desired setpoint while collecting experimental data for analysis. Step response experiments demonstrated the behavior of the control system and allowed evaluation of controller performance.

The modular architecture developed in this lab provides a scalable foundation for future robotics applications, including trajectory tracking, autonomous navigation, and sensor fusion.
